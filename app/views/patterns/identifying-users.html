Identifying users

Patterns to help services identify their users.

  • Usage: internal/external
  • Status: experimental
  • Last updated: 28 November 2017

Authenticating users

This lets the system know that the user who did something earlier is the same user now. An example would be a user starting an application is the same user who returned to finish it later.

Authentication does not:

  • confirm who the user is
  • what the user can do
  • if the user is who they say they are

Identity assurance

Identity assurance confirms the true identity of a user. The checks are usually done before confirming a users suitability for something. The assurance can consist of:

  • confirming the user is a human
  • confirming the validaty of information and documents
  • fraud checks
  • checking for warnings about a user

User account

We should avoid user accounts where possible. Making users create accounts when not needed:

  • forces users to keep track of more logins then neccesary
  • creates a security risk by storing users data
  • can grow the amount of redundant databases - which is expensive and a security risk

Only use user accounts when:

  • users need to use a service frequently
  • you need to audit user behaviour or collect data about users

Guidance

Considerations you should make:

  • balance frequency of use, security requirements and convenience of the user
  • think about security trade-offs and data storage early in the design of a service
  • consult specilists
