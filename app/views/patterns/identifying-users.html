Authentication

Let users access your service in a secure way.

  • Usage: internal/external
  • Status: experimental
  • Last updated: 18 February 2018

When to use authentication

Authentication lets users verify who they are.

Most one-time services won't need to authenticate their users. When a service does need to verify a user’s identity, it needs to balance ease of access with keeping a service and its user data secure. You should think about this early on when designing your service.

You might want to authenticate users if:

  • your service allows users to save and complete a transaction at another time
  • a user can check the status of a transaction
  • a user will use a service multiple times

Authentication examples

You can authenticate users using reference numbers, personal details, codes sent in SMS and emails, and by using a third-party. Don’t use National Insurance numbers to verify a user’s identity.

You should consult a Home Office security specialist as they can advise you on the type of authentication that’s right for your service.

Authentication is not the same as identity assurance

Identity assurance confirms the true identity of a user. The checks are usually done before confirming a users suitability for something. Identity assurance can consist of:

  • confirming the user is a human
  • confirming the validaty of information and documents
  • fraud checks
  • checking for warnings about a user

User accounts

User accounts are a form of authentication that you should try to avoid. Services should be public facing and not start behind a user log in. This is essential for users searching for the government service they need. Making users create accounts when not needed:

  • creates a barrier for users and can reduce completion rates
  • forces users to keep track of more logins then neccesary
  • burdens teams with databases that are difficult to build and maintain
  • increases the amount of redundant databases - which is expensive and a security risk

Only make users create an account if:

  • users access a service frequently
  • users need it - don't force them to create an account until necessary
  • a service needs to audit user behaviour
