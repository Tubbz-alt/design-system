Why have we created this pattern?

To define buttons types and when to use each type. We have included secondary buttons, for use on admin interfaces. Removing the affordance style from disabled buttons as well as adding opacity to show visually it can not be pressed.

Interaction principles

  • buttons should look like buttons
  • buttons need to have a good level of affordance, so that users know that something meaningful will happen if they click a button
  • buttons can have an icon to further support their action or state
  • don't make things that aren't buttons look like buttons
  • avoid secondary actions unless you see a need for one in research
  • primary actions should be aligned with input fields to increase completion rates

Content principles

  • label buttons with what they do

Accessibility principles

  • buttons should have clear hover and focus states
  • use a button tag with a role='button'

User research

  • users sometimes confuse grey buttons with disabled buttons
  • users sometimes confuse white buttons with input fields
  • white buttons worked better in confirmation pages and interruption cards
