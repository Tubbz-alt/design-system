Buttons

Use a button when a user needs to take an action, submit data, or make a change to the page.

Why have we created this pattern?

The GOV.UK pattern does not include secondary actions or disabled buttons, which may be required on admin interfaces.

Interaction principles

Buttons should look like buttons.

Don’t make things that aren’t buttons look like buttons.

So that users know something meaningful will happen if they click, buttons need to have a good level of affordance.

Buttons can have an icon to further support their action or state.

Avoid secondary actions unless research has identified a need for one.

Primary actions should be aligned with input fields to increase completion rates.

Content principles

Label buttons with the outcome of the action: "Complete application" rather than "OK".

Be particularly careful when labelling "Cancel" buttons that the action is clear.

Accessibility principles

Buttons should have clear hover and focus states.

Use a button tag with role='button'.

User research

Users sometimes confuse grey buttons with disabled buttons.

Users sometimes confuse white buttons with input fields.

White buttons have worked better in confirmation pages and interruption cards.
